📋 Table Reference

👤 User

Stores authenticated users. Each user has a unique sub (subject identifier from OIDC) and email. The scope field holds space-separated permissions: dabih:base dabih:api dabih:admin

🔑 PublicKey

RSA public keys belonging to users. Keys start disabled (pending approval). Once enabled, they're used to encrypt file AES keys and sign refresh tokens. isRootKey marks organization-wide recovery keys.

📁 Inode

Virtual filesystem node — can be a FILE, DIRECTORY, UPLOAD (in-progress), TRASH, ROOT, or HOME. Uses human-memorable mnemonic identifiers. Self-referencing parentId creates the directory tree.

💾 FileData

Stores file metadata. uid is the storage bucket identifier. keyHash is the SHA-256 of the AES encryption key. size is null during upload and set when finalized.

🧩 Chunk

Individual encrypted pieces of a file. Each chunk has its own AES iv, byte range (start–end), SHA-256 hash for integrity, and crc for error detection.

👥 Member

Maps users to inodes with permissions: 0=NONE 1=READ 2=WRITE. Unique constraint on (sub, inodeId) prevents duplicates.

🔐 Key

Stores the file's AES encryption key, RSA-encrypted with a user's public key. Each file has one Key entry per authorized public key, enabling multi-user decryption.

🎫 Token

Long-lived API access tokens (alternative to JWT). Used by CLI and automation. Scoped permissions with optional expiration. Stored in DB for instant revocation.
