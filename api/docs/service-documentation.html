Core Concepts & Data ModelsUnderstanding the fundamental building blocks of Dabih

🌲 What is an Inode?

An Inode (Index Node) is the fundamental unit of the virtual filesystem in Dabih, borrowed from Unix/Linux filesystem design. Every file, directory, trash bin, and user home folder is represented as an Inode record in the database.

Think of it like this: In a real computer, every file and folder has an inode number. Dabih recreates this concept in a database to build a virtual filesystem over encrypted cloud storage.

📋 Inode Types

  • type=0 → FILE — A completed, encrypted file. Has associated FileData and Chunks.
  • type=1 → DIRECTORY — A folder that contains other inodes as children.
  • type=2 → UPLOAD — A file that is currently being uploaded. Transitions to FILE when all chunks are uploaded and verified.
  • type=10 → TRASH — A special directory (recycle bin). Each user's HOME has one TRASH folder. When files are "deleted", they're moved here first.
  • type=11 → ROOT — The single global root of the entire filesystem tree. Created automatically on first server boot. All HOME directories are children of ROOT.
  • type=12 → HOME — Each user's personal root directory. Created automatically when a user registers. Named after the user's email.

Every Inode has a unique mnemonic — a human-readable identifier (like alpha-bravo-charlie) used in URLs instead of database IDs. This makes links shareable and memorable.

🧩 What is a Chunk?

A Chunk is a piece of an encrypted file. Large files are split into multiple chunks during upload, with each chunk independently encrypted using AES-256-CBC with a unique initialization vector (IV).

Why chunked storage?

① Resumable uploads: If a large upload fails, only the failed chunk needs to be re-uploaded.
② Integrity verification: Each chunk has its own SHA-256 hash and CRC-32 checksum. A single corrupted chunk can be identified without re-downloading the entire file.
③ Streaming: Files can be downloaded and decrypted chunk by chunk, without needing the entire file in memory.

Each chunk stores: its SHA-256 hash (of the plaintext), the AES iv used for encryption, the byte range (start–end), and the CRC-32 crc of the encrypted output.

💾 What is FileData?

A FileData record holds the metadata for the actual file content — separate from the Inode (which represents the file's position in the filesystem). This separation allows the same data to be referenced by multiple Inodes (deduplication potential).

Key fields:

  • uid — A 12-character identifier that maps to a storage bucket (folder on disk). This is where the encrypted chunk files are physically stored.
  • keyHash — The SHA-256 hash of the AES encryption key. Used to verify that a client-provided decryption key is correct without revealing the key itself.
  • hash — A composite SHA-256 hash of all chunk hashes. Set when the upload is finalized. Serves as a file fingerprint.
  • size — Total file size in bytes. Null during upload, set on finish. Used to validate chunk completeness.

👥 What is a Member?

A Member record establishes an access control link between a user and an Inode. It defines what permission a user has on a specific file or directory.

Permission Levels

0 = NONE — No access (explicitly denied).
1 = READ — Can view/download the file or list directory contents.
2 = WRITE — Full access: read, upload, share, edit, and delete.

Permissions propagate down the tree: if you have WRITE on a directory, you have WRITE on all files and subdirectories inside it. A Member record on a parent directory grants access to all descendants.

🔐 What is a Key (Encryption Key Record)?

A Key record stores the AES encryption key for a file, encrypted with an RSA public key. Each file has one Key record per authorized public key.

How the encryption key lifecycle works:

① Upload start: Server generates a random AES-256 key → encrypts it with each authorized RSA public key → stores as Key records → stores plaintext key in Redis for the upload duration.
② Upload finish: Ephemeral key deleted from Redis. Only the RSA-encrypted copies remain.
③ Download: Client fetches Key record → decrypts AES key using their RSA private key (client-side) → sends AES key to server → server uses it to decrypt the file stream.

This design means: the server never permanently stores the plaintext encryption key. Only users with the matching RSA private key can decrypt files, achieving true end-to-end encryption.

🔑 What is a PublicKey?

A PublicKey is an RSA public key registered by a user. It serves two purposes:

  • Encryption: When a file is uploaded, the AES key is encrypted with every active PublicKey, creating a Key record for each. This allows multiple users to decrypt the same file.
  • Authentication: Users can sign refresh tokens with their RSA private key. The server verifies the signature against their stored public key.

Keys start in a disabled state (enabled=null). An authorized user must explicitly enable them. This prevents unauthorized key injection — you can't just register a key and start decrypting files.

Root Keys (isRootKey=true) are special: they encrypt every upload, providing organizational-level data recovery capability.

🎫 What is a Token?

A Token is a long-lived, database-stored access credential (as opposed to a JWT, which is stateless). Tokens are used for:

  • CLI/Automation: Instead of managing JWT refresh flows, CLI tools use a persistent token.
  • Scoped download: Short-lived tokens (30s) with scope dabih:download:<mnemonic> for secure file streaming.

Because tokens live in the database, they can be instantly revoked (unlike JWTs, which remain valid until expiration).

🧠 Scopes

The authorization system uses three scopes:

Scope Meaning Granted When
dabih:base Basic sign-in access. Can view profile, add public keys. Always granted to all users
dabih:api Full API access. All file/folder/user operations. When user has at least one enabled PublicKey
dabih:admin Administrative access. User management, data integrity. First user gets it automatically; others via admin grant

Auth Servicesrc/api/auth/ · Route: /api/v1/auth · 7 files

What it is: The authentication and authorization gateway. Handles user login via two mechanisms — OpenID Connect (OIDC) for SSO and email magic links for standalone deployments. Issues JWTs for session management and supports token refresh via client-signed RSA tokens.

📄 Files in this Service

File Purpose
controller.ts TSOA controller defining all auth endpoints with decorators
login.ts Generates PKCE code verifier + state, stores in Redis, returns OIDC redirect URL
callback.ts Handles OIDC callback: exchanges code for token, fetches userinfo, creates/updates user, signs JWT
signIn.ts Email magic link flow: generates email JWT, sends via Resend or returns token directly if email is not configured
verifyEmail.ts Validates email verification token, creates user if new, creates home directory, returns auth JWT
refresh.ts Verifies client-signed RSA token against stored public keys, checks 2-week lastAuthAt window, issues fresh JWT
auth.test.ts Integration tests for the auth flow

🔧 How It's Configured

  • OIDC_PROVIDER — Identity provider URL (e.g., Google, GitHub, Keycloak). If not set, OIDC login is disabled.
  • OIDC_CLIENT_ID / OIDC_CLIENT_SECRET — OAuth2 credentials from your OIDC provider.
  • RESEND_TOKEN / RESEND_EMAIL — If set, enables email magic links. If not set, tokens are returned directly (dev mode).
  • SECRET — Master secret used to derive AES keys for encrypting JWT signing secrets in Redis.

🏗️ Why This Architecture?

  • Dual auth — OIDC for institutions with existing identity providers; email for standalone deployments without SSO infrastructure.
  • PKCE flow — Proof Key for Code Exchange prevents authorization code interception. Code verifiers stored in Redis (10 min TTL) for stateless server design.
  • Rotating secrets — JWT signing secrets rotate daily and are stored encrypted in Redis (5-deep list). Old secrets remain valid for verification during rotation, preventing sudden invalidation.
  • RSA refresh — Requiring client-signed RSA tokens for refresh ensures the client still possesses their private key, not just a stolen JWT.
  • Auto-admin — First registered user automatically gets admin scope, bootstrapping the system without manual database intervention.
  • Rate limiting — Sign-in endpoint is rate-limited (3 requests per minute per IP) to prevent email spam abuse.

Upload Servicesrc/api/upload/ · Route: /api/v1/upload · 10 files

What it is: Manages the complete lifecycle of encrypted file uploads — from initialization through chunked data transfer to finalization. Each file is encrypted with a fresh AES-256 key, split into chunks, and stored independently on the storage backend.

📄 Files in this Service

File Purpose
controller.ts TSOA controller with endpoints: start, chunk, stream, finish, cancel, unfinished, cleanup
start.ts Generates AES key, creates FileData + Inode records, encrypts AES key with RSA public keys, stores in Redis + creates storage bucket
chunk.ts Receives raw bytes → validates hash → encrypts → computes CRC-32 → writes to storage → creates Chunk record. Supports both application/octet-stream and multipart form upload
stream.ts Single-request upload: combines start + chunk + finish into one streaming PUT request
finish.ts Validates all chunks are contiguous (no gaps), computes composite hash, transitions Inode type UPLOAD→FILE, deletes ephemeral key
cancel.ts Removes inode, file data, chunks, and storage bucket for an in-progress upload
unfinished.ts Lists all uploads that haven't been finalized (type=UPLOAD)
cleanup.ts Batch removes all unfinished uploads for the current user
util.ts Parses Content-Range and Digest HTTP headers
upload.test.ts Integration tests

🔧 How the Encryption Works (Step by Step)

  • Step 1 — Server generates a random AES-256 key for the file
  • Step 2 — The SHA-256 hash of this key is stored in FileData.keyHash (used later to verify client-supplied keys)
  • Step 3 — The AES key is encrypted with every authorized RSA public key (user's keys + root keys) → stored as Key records
  • Step 4 — The plaintext AES key is cached in Redis (aesKey:{sub}:{mnemonic}, 1hr TTL) for the upload duration
  • Step 5 — For each chunk: Raw bytes → SHA-256 validate → AES-256-CBC encrypt (unique IV per chunk) → CRC-32 → write to disk
  • Step 6 — On finish: validate chunks, delete ephemeral key from Redis. Only RSA-encrypted copies remain.

🏗️ Why This Architecture?

  • Chunked transfer — Enables large file support (multi-GB) with built-in resume capability
  • Per-chunk encryption with unique IV — Prevents pattern analysis attacks that could occur with a single IV
  • Hash verification — SHA-256 digest header vs actual data ensures integrity during transit
  • Ephemeral key in Redis — The server only holds the plaintext key during active upload. If the server is compromised after upload, the key is gone.
  • Multi-key encryption — Encrypting the AES key with multiple RSA public keys allows any authorized user to decrypt without sharing private keys

Download Servicesrc/api/download/ · Route: /api/v1/download · 5 files

What it is: Handles decryption and streaming of encrypted files. Uses a two-step process: first the client provides the AES key (decrypted client-side), then the server uses it to stream the decrypted file. Scoped, short-lived tokens ensure security.

📄 Files in this Service

File Purpose
controller.ts TSOA controller: decrypt, download (stream), chunk (raw)
decrypt.ts Validates AES key hash → stores key in Redis → creates scoped DB token (30s TTL) with scope dabih:download:{mnemonic}
mnemonic.ts Validates scoped token → reads AES key from Redis → streams all chunks through AES decrypt pipeline → assembles full file
chunk.ts Direct raw chunk download (encrypted binary, no decryption)
download.test.ts Integration tests

🏗️ Why Two Steps?

  • Security separation — The decrypt step (POST) accepts the AES key and returns a minimal, scoped token. The download step (GET) uses this token to stream the file. This prevents the key from being logged in URL query parameters.
  • 30-second token TTL — The download token expires quickly, preventing replay attacks or token theft
  • Client-side RSA decryption — The server never sees the RSA private key. The AES key is decrypted in the browser/CLI, then sent to the server only when needed.

Filesystem Servicesrc/api/fs/ · Route: /api/v1/fs · 21 files (largest service)

What it is: A virtual filesystem providing directory management, file organization, member sharing, access control, and search. Built on the Inode tree structure, it supports CRUD operations on directories, file listing, tree traversal, duplication, moving, sharing, and async search.

📄 Key Files

File What It Does
list.ts Lists children of a directory inode (or home if none specified). Returns inode details + members + file data
listShared.ts Lists directories shared with the current user (where they're a member but not the owner)
file.ts Gets file info needed for download: inode, file data, chunks, and encrypted keys
listFiles.ts Recursively lists all FILE-type inodes in a directory tree
listParents.ts Walks up the parent chain from an inode to root (for breadcrumb navigation)
addDirectory.ts Creates a new DIRECTORY inode under a parent, generates mnemonic
move.ts Moves an inode to a new parent directory (validates write permission on both source and target)
duplicate.ts Deep copies an inode and all its children, generating new mnemonics
remove.ts Moves an inode to the user's TRASH directory (soft delete)
destroy.ts Permanently deletes an inode and all associated data (hard delete)
addMembers.ts Shares a directory/file with other users by creating Member records
setAccess.ts Updates a member's permission level on an inode
resolve.ts Resolves a path string (like /Projects/Data) to an inode
tree.ts Returns the full tree structure under an inode
searchStart.ts Starts a background search job (Piscina worker thread)
searchResults.ts Polls search results from Redis job data
searchCancel.ts Cancels a running search job

🏗️ Why This Architecture?

  • Unix-like inode tree — Familiar concepts (home directory, trash bin, tree hierarchy) make the API intuitive and the file organization natural
  • Mnemonics — Human-readable identifiers replace raw database IDs. Easier to debug, share, and use in URLs
  • Async search via Piscina — Filesystem search can be slow (BFS traversal). Offloading to a worker thread prevents blocking the main event loop. Results stream into Redis, and the client polls via the Job API
  • Member-based ACL — Simple but effective: each user-inode pair has a permission level. Inheritance through the tree hierarchy means sharing a directory automatically shares all its contents
  • Two-phase delete — remove moves to TRASH (recoverable), destroy permanently deletes. Reduces accidental data loss

User Servicesrc/api/user/ · Route: /api/v1/user · 12 files

What it is: Manages user accounts and their RSA public keys. Users register, manage their cryptographic keys (required for file encryption/decryption), and admins can promote/demote other users. The key management here is the foundation of Dabih's end-to-end encryption.

📄 Key Files

File What It Does
add.ts Registers a new user (admin-only)
get.ts Fetches user profile by sub, including their public keys
list.ts Lists all users in the system
remove.ts Removes a user account
addKey.ts Registers an RSA public key for the current user. Key is created in disabled state (pending approval).
enableKey.ts Enables a pending public key. Once enabled, the user gains dabih:api scope.
removeKey.ts Removes a public key
findKey.ts Finds which user owns a particular public key by hash (no auth required)
setAdmin.ts Grants or revokes admin scope for a user (admin-only)
util.ts User utility functions

🏗️ Why This Architecture?

  • Key approval workflow — Keys start disabled to prevent unauthorized key injection. A trusted user must explicitly enable them. This ensures only legitimate keys participate in file encryption.
  • Root keys — Organization-level recovery mechanism. Root keys encrypt every upload, so even if a user loses their private key, an admin with the root private key can still decrypt the data.
  • Scope-based access escalation — Users start with dabih:base (can only sign in and add keys). They get dabih:api when a key is enabled. This two-step process prevents users from accessing the API without proper cryptographic setup.

Token Servicesrc/api/token/ · Route: /api/v1/token · 5 files

What it is: Manages persistent API access tokens stored in the database. These complement JWTs — while JWTs are short-lived for browser sessions, DB tokens are long-lived for CLI tools, scripts, and integrations. They can be instantly revoked.

📄 Files

File Purpose
add.ts Creates a new token with specified scope and optional expiration
list.ts Lists all tokens belonging to the current user
remove.ts Revokes (deletes) a token

🏗️ Why DB Tokens + JWTs?

  • JWTs are fast (no DB lookup) but can't be revoked before expiration → perfect for browser sessions
  • DB tokens require a DB lookup but can be instantly revoked → perfect for CLI/automation where long-lived access is needed
  • The auth layer checks the token format to determine which type it is (isToken()) and handles each appropriately

FileData Service (Admin Only)src/api/filedata/ · Route: /api/v1/filedata · 4 files

What it is: Administrative maintenance endpoints for data integrity management. Allows admins to find orphaned file data (data not referenced by any inode), remove it, and cross-check the database against the storage backend for inconsistencies.

📄 Files

File Purpose
orphaned.ts Queries FileData records that have zero Inode references (data exists but no file points to it)
remove.ts Deletes a FileData record, its chunks, and the physical storage bucket. Uses Prisma transaction.
checkIntegrity.ts Cross-references all FileData UIDs against actual storage buckets. Returns missing (in DB but not on disk) and unknown (on disk but not in DB)

🏗️ Why?

  • Orphaned data occurs when Inodes are deleted/destroyed but the underlying FileData is retained (or vice versa during crashes)
  • Integrity checks catch storage/DB mismatches that could indicate data corruption, incomplete cleanup, or manual filesystem tampering

Job Service (Admin Only)src/api/job/ · Route: /api/v1/job · 2 files

What it is: Administrative endpoint to monitor background jobs. Jobs are stored entirely in Redis (metadata as hash, results as list). Currently used exclusively by the filesystem search system — each search query creates a job that runs in a Piscina worker thread.

How Jobs Work

  • Job created → job:{id}:meta hash in Redis with status=running
  • Worker adds results → job:{id}:data list in Redis (appended as JSON strings)
  • Client polls results → reads up to 100 entries at a time, then removes them (consumer pattern)
  • Job completes → status=complete. Client stops polling.
  • Job cancelled → status changed, worker checks on each iteration and exits early
