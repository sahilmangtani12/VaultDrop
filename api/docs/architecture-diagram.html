Part 1: High-Level System Overview How all components connect together

The system consists of a Koa.js HTTP server with auto-generated TSOA routes, backed by PostgreSQL (via Prisma ORM) and Redis for ephemeral state. Files are stored encrypted on a pluggable storage backend. Authentication is handled via OpenID Connect (SSO) or email magic links (Resend).

Part 2: Authentication Flows OIDC Login · Email Magic Link · Token Refresh

Dabih supports dual authentication: OpenID Connect with PKCE for institutional SSO, and email magic links for standalone deployments. Both flows produce a JWT token signed with rotating secrets stored in Redis. Long-lived DB tokens serve CLI/automation.

Part 3: Encrypted Upload & Download Pipeline End-to-end encryption with chunked storage

Files are split into chunks, each independently encrypted with AES-256-CBC. The AES key is encrypted with each authorized user's RSA public key and stored in the Key table. Only the key owner can decrypt. The server holds plaintext keys only ephemerally in Redis during active upload/download sessions.

Part 4: Virtual Filesystem Structure Inode tree hierarchy with permissions

The filesystem uses a Unix-like inode tree. A single ROOT inode contains HOME directories per user. Each HOME has a TRASH bin. Files and directories are identified by human-memorable mnemonics. Access is controlled through Member records with READ/WRITE permissions.

Part 5: Redis Ephemeral State Management All transient data lives in Redis with automatic TTL expiration

Redis serves as the ephemeral state layer. Rotating JWT signing secrets, temporary AES keys, PKCE verifiers, rate limits, and background job results all live here with natural TTL-based expiration — keeping PostgreSQL clean and focused on persistent data.
